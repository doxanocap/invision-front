# inVision — AI-Assisted Candidate Evaluation System

> Помогаем приёмной комиссии быстрее и точнее увидеть потенциал кандидата — без чёрного ящика.

---

## Одна проблема, которую мы решаем

Комиссия боится пропустить сильного кандидата — но не может объяснить, кто сильный, потому что чётких критериев нет.

Это ведёт к трём системным последствиям:
- решения **субъективны** и зависят от конкретного человека, а не от качества кандидата
- процесс **не масштабируется** — знание не передаётся между оценщиками
- выбор **нельзя защитить** — нет прозрачного обоснования для отказа или приёма

---

## Шесть реальных болей процесса

| # | Боль |
|---|------|
| 1 | Усталость оценщика меняет решения по ходу дня |
| 2 | Знание не передаётся между людьми и циклами отбора |
| 3 | Красивый текст побеждает реальный потенциал |
| 4 | Расхождение между оценщиками — без объяснения |
| 5 | Нет обоснования для отказа |
| 6 | Спорный кандидат теряется в процессе |

---

## Что продаёт этот продукт

**Страх пропустить сильного кандидата.**

Не скорость. Не автоматизацию. Не ИИ ради ИИ.
Уверенность комиссии в том, что решение обосновано — и его можно объяснить.

---

## Принципы, которые не нарушаем

- **ИИ не принимает финальное решение** — всегда human-in-the-loop
- **Никакой демографии** как критерия оценки
- **Explainability обязательна** — каждый балл объяснён
- **Система работает без AI** — LLM ускоряет, структурирует и объясняет, но не заменяет логику

---

## Пайплайн (7 шагов)

```
Кандидат → Анкета → Скрининг → Кейс-вопрос → Оценка операторами → Видео → Authenticity Delta → Финальная карточка → Решение комиссии
```

### Шаг 1 — Анкета (Кандидат)
Кандидат заполняет форму: базовые данные, IELTS/UNT балл, BIO, CV.  
AI **только структурирует** и флагирует пропущенные поля. Не скорит.

### Шаг 2 — Скрининг (AI)
AI проверяет минимальные пороги (IELTS ≥ X, UNT ≥ Y, BIO содержательно).  
Результат: **PASS / REVIEW / FAIL**.  
FAIL — автоматически. REVIEW — флаг для ручного просмотра. PASS — идёт дальше.  
> Без системы: человек читает каждую анкету. С системой: человек смотрит только REVIEW.

### Шаг 3 — Кейс-вопрос (Кандидат)
Один вопрос о лидерстве, текстовый ответ 300–500 слов, лимит времени.  
AI структурирует ответ по трём критериям:
1. Инициатива без разрешения
2. Рост через провал
3. Влияние на других

### Шаг 4 — Оценка кейса (Два оператора)
Оператор 1 и Оператор 2 **независимо** читают структурированный ответ и ставят оценку 1–5 по каждому критерию + одно предложение-обоснование.  
AI после обоих оценок: нормализует, флагирует расхождение > 2 баллов как «спорный», объясняет разницу.

### Шаг 5 — Видео-ответ (Кандидат)
Кандидат записывает 1–2 минуты видео по тому же кейсу.  
Whisper (локально) транскрибирует видео → текст.

### Шаг 6 — Authenticity Delta (AI)
Сравнивает текстовый ответ (Шаг 3) и транскрипт видео (Шаг 5):

| Сигнал | Интерпретация |
|--------|---------------|
| Высокое сходство | Подозрение — заучил или скопировал |
| Естественная разница | Сигнал аутентичности |
| Полное несовпадение | Флаг — вероятно, другой человек писал |

Визуально: gauge 🟢 / 🟡 / 🔴. Одна строка: «Кандидат владеет своим ответом / не владеет».

### Шаг 7 — Финальная карточка (Комиссия)
Один экран. Вся информация. Комиссия видит:
- Имя, IELTS, UNT, статус скрининга
- Радарная диаграмма: Инициатива / Рост / Влияние
- Ключевые цитаты из кейса (3 фразы)
- Authenticity Delta
- AI Summary — 2–3 предложения о кандидате
- Флаги (если есть)
- Кнопки: **[В ШОРТЛИСТ] [ОБСУДИТЬ] [ОТКАЗ]**

Это и есть human-in-the-loop — явно, визуально, неизбежно.

## Четыре задачи LLM — и только они

| Задача | Что делает |
|--------|------------|
| Структурирование | Хаос → читаемый формат |
| Скоринг по рубрике | Ответ + критерий → оценка |
| Нормализация | Два независимых мнения → единая шкала |
| Объяснение | Скор → «почему именно столько» |

Всё за пределами этих четырёх — человек или простая детерминированная логика.

---

## Критерии успеха (валидация без ML)

- Комиссия принимает решение **быстрее** при наличии структурированной карточки
- **Уверенность** в решении выше (самооценка по шкале 1–5)
- **Аргументация** конкретнее — оценщики ссылаются на факты, а не ощущения
- Спорные кандидаты **не теряются** — флагируются для обсуждения

## Что намеренно не строим (MVP)

- GPT-детекция плагиата — roadmap
- Красивый UI в ущерб логике — последний приоритет
- Автоматическое финальное решение — никогда
- Демографические сигналы как критерии — никогда

## Config

Основа конфигурации — `lib/config.ts`, который берет значения из переменных.

Создайте локальный конфиг на основе примера:
```bash
cp .env.example .env.local
```

Основные переменные:
- `API_BASE_URL` — базовый URL бэкенда (например, `http://192.168.1.109:8000`). Все запросы прозрачно используют эту переменную.

## Quickstart

Для быстрого старта выполните:
```bash
make env && make install && make dev
```

## Docker

Запуск приложения одной командой:
```bash
docker compose up --build
```
